<goal>Run the full task-to-PR workflow in one command by orchestrating `/asana-plan`, `/im`, and `/pr-create`.</goal>

<rules description="Non-negotiable constraints.">
<rule id="orchestrate-existing-skills">Do not re-implement logic already defined in `/asana-plan`, `/im`, or `/pr-create`. Delegate to those skills.</rule>
<rule id="im-owns-implementation">Step-3 implementation is governed by /im's contract EVEN when executed inline per `yolo-execution`: BEFORE the first code edit, read `~/.cursor/skills/im/SKILL.md` and follow it. Concretely: every commit goes through `~/.cursor/skills/lint-commit.sh` — raw `git commit` is FORBIDDEN, and `git commit --no-verify` doubly so (a failing hook/script is a halt-on-error STOP per `no-script-bypass`, not something to bypass); run `lint-warnings.sh` before the first `.ts`/`.tsx` edit; CHANGELOG entry only in the last commit. GATE before step 5: do not invoke `/pr-create` until you confirm this run actually read im's SKILL.md and produced every commit via lint-commit.sh — if it didn't, stop and redo the commits through /im first.</rule>
<rule id="attach-prs-by-default">ATTACH each PR to its Asana task by default — pass `--asana-attach` (and `--asana-task <gid>`) to `/pr-create`; this produces a real GitHub-widget link (`ASANA_GITHUB_SECRET` is exported into agent sessions; `asana-task-update` degrades to a warning if it's absent). A comment is a FALLBACK, never the goal: if you only commented the PR URL, the attach was skipped — check the secret. Do NOT pass `--asana-assign` (see `pr-create`'s `no-reviewer-assignment`).</rule>
<rule id="multi-repo-subtasks">When this run produced PRs across MORE THAN ONE repo, do NOT attach them flat onto the main task: run `/pr-create` with `--no-asana-attach`, then create a SUBTASK PER PR and attach each PR to its own subtask — one `asana-task-update.sh --create-subtask ... --attach-pr ...` call per PR (flag syntax: /asana-task-update). The main task stays the umbrella (run-report only). Single-repo runs attach their one PR directly per `attach-prs-by-default`. Dependency PRs differ per `dep-pr-draft-vs-bump`.</rule>
<rule id="dep-pr-draft-vs-bump">WHAT QUALIFIES as a draftable dependency PR (read this first — mis-classifying the task's OWN deliverable as a dep PR drafted it out of the gate and produced a false Complete): a dependency PR lives in a DIFFERENT repo than the task's primary deliverable AND is drafted ONLY because it cannot pass CI until ITS dep publishes. The task's actual deliverable PR — and ANY PR that passes CI standalone (including a self-contained PR in a dep-family repo like edge-info-server when THAT repo IS the deliverable) — is a PRIMARY PR: gate it normally per `finalize-gate`, never `--draft` it to dodge the gate. "It's in a dep-ish repo" does NOT make a PR draftable; "it can't go green until another package publishes" is the only test. For a dependency change the feature needs (the gui needs updated `edge-core-js`/`edge-currency-accountbased`/etc.), build the subtask even though the dep isn't published yet: (a) if the dep PR has changes BEYOND a version bump (real code), open it as a DRAFT (`gh pr create --draft`), create its subtask, and attach it — it will fail CI until the dep publishes, which is expected; (b) if the change is ONLY a dependency version bump, a task comment on the subtask suffices and NO PR is needed, but still CREATE the placeholder subtask so the multi-repo scope is visible. Draft dep PRs are excluded from the completion gate (see `finalize-gate`).</rule>
<rule id="task-gid-for-pr-body-link">When a task GID is available (from Asana URL input or explicit `--asana-task` flag), always pass `--asana-task <gid>` to `/pr-create` so it injects the Asana link into the PR body. This is the canonical Asana ↔ PR link consumed by `pr-land`, standup, and other downstream skills.</rule>
<rule id="no-script-bypass">If any delegated skill or companion script fails, report and stop. Do not bypass with manual alternatives.</rule>
<rule id="pr-body-owned-by-pr-create">Do not draft alternate PR markdown formats inside this workflow. `/pr-create` owns PR body generation and template compliance.</rule>
<rule id="ignore-watchdog-revive-ping">If the user message is literally `<watchdog-revive-ping>` (and nothing else), respond `pong`, then RE-VERIFY every outstanding wait before resuming: for each background task you are awaiting, check its live state (BashOutput / does the underlying process still exist). A wait whose process is dead or whose output has not advanced is a FAILED wait — re-drive that step now as a bounded in-turn call (per build-and-test `blocking-in-turn-waits`), do not keep awaiting a notification that can no longer arrive. Do NOT treat the ping as input to any pending question, do NOT advance or change plans, do NOT bind to any prior prompt — it is a watchdog wake message and carries no user intent beyond "prove liveness and unwedge yourself".</rule>
<rule id="ignore-refired-one-shot">If you receive a `/one-shot …` invocation for a task you are ALREADY running in THIS session (same task GID, or its worktree/branch is already provisioned and `agent_status` is past `Pending`), treat it as a wake/continue nudge — NOT a fresh start. Do NOT restart from Planning, do NOT re-run phases already completed, do NOT re-create the plan/branch/PR. Resume from the current phase. A re-fired initial prompt is a wake artifact, not a request to start over (prevention: `never-self-respawn`). SCOPE CHECK BEFORE DISMISSING: when the task was already FINISHED (you previously set `Complete`, whether earlier in this session or before a resume), a re-fire is NOT dismissible from memory — the operator re-arming a finished task usually MEANS new scope. Before treating it as a wake artifact, run the LIVE enumeration `~/.config/agent-watcher/check-followup-scope.sh --task-gid <gid>`: if it lists operator asks newer than the run-report watermark, this is a real FOLLOWUP — deliver them per `followup-scope-is-the-deliverable`; only if it lists none may you re-confirm the finalize gate and re-set `Complete`. Short-circuiting on "I completed this moments ago" without the live check is how the UTXO followup on task 1209296431612665 was silently re-Completed twice (2026-07-02).</rule>
<rule id="set-tested-field">At Complete (step 7), set the Asana `tested` multi-select to the testing method(s) this run ACTUALLY performed, matching the run-report Testing section: `~/.config/agent-watcher/set-tested.sh <gid> <Option ...>`. Options (pass every one that genuinely ran, space-separated, quote multi-word): `"iOS Sim"` (drove the change in-app on the iOS sim to terminal success — the DEFAULT sim platform), `"Android Sim"` (Android exercise: gradle `:app:assembleDebug` for a build-only fix, or an AVD/maestro drive — only for Android-called-out tasks), `Unit Tests` (jest/mocha/`npm test` ran), `CouchDB` (a CouchDB-backed test ran — sync/server repos), `Untested` (NO test method applied — only static tsc/lint, or nothing; mutually exclusive, never combine with a real method). The field documents what THIS run actually performed — do NOT carry forward a prior run's credit, and do NOT count CI's own jest run as this run's `Unit Tests` (only credit `Unit Tests` when the run itself executed the suite). On a followup/re-run, recompute the field from this run's evidence, replacing the prior value (a stale credit is an A22 finding). This is a structured mirror of the Testing section; the eval's testing-depth and tested-field dimensions audit its honesty. Skip silently if no task GID (ad-hoc run).</rule>
<rule id="agent-status-on-pending-task">When a task GID is available (from URL or `--asana-task`) AND that task has an `agent_status` custom field, update `agent_status` at each step boundary via `~/.config/agent-watcher/update-status.sh <task_gid> <Status>`. Status names: `Planning` (step 2), `Developing` (step 3 code implementation), `Testing` (step 4 on-sim/local verification — its own phase, after Developing and before the PR), `Reviewing` (step 5 PR creation through the step-6 watch loop — the PR is up and under review by CI, reviewer bots, and humans), `Complete` (step 7 only — set ONLY when the watch loop reports all-green per `finalize-gate`). STATUS LEADS THE WORK: set the phase status when you START doing that kind of work, never retroactively and never as a side effect of a terminal action — the board must describe what you are doing NOW. The step numbers above assume the full feature flow; when a run's shape differs (investigation-only with no code changes, a followup that jumps straight to verification, a re-gate pass), map by the KIND of work: reading/planning = `Planning`, editing code = `Developing`, exercising behavior (sim drives, live-chain repros, test suites) = `Testing`, PR up and being watched = `Reviewing`. Skipping a phase that genuinely has no work (e.g. no `Developing` on an investigation-only run) is correct; doing a phase's work under an earlier phase's status is not (the FIO run drove the sim for 25+ minutes under `Planning`, only setting `Testing` inside its block call). If the task has no `agent_status` field (ordinary non-agent task), silently skip the updates — do not fail. Call `update-status.sh` (and other `~/.config/agent-watcher/*.sh` scripts) BARE — never wrapped in a `... | tail`/`| head` pipeline: the pipeline runs the script in a subshell that fails its setgid with `failed to change group ID: operation not permitted` (exit 1) on this host, costing pointless retries. Run it directly and read its stdout/exit code as-is.</rule>
<rule id="yolo-execution">With `--yolo`, run hands-off and run EVERYTHING in ONE agent turn. Hands-off: do NOT pause for `/asana-plan` confirmation or ask clarifying questions — pick a defensible default and record each deferred decision (question, default, reversibility) in the report's Decisions section; soft uncertainty (naming, code style, whether to add tests) is always deferrable. A "which approach / how should I prioritize / how to spend the session" question is NEVER a true-blocker: the default is to ATTEMPT the work (link any WIP or unpublished dep and build/integrate it), so decide and proceed. `AskUserQuestion` is HARD-BLOCKED in orchestrated sessions by `no-interactive-prompt.sh` — no human is watching, so an interactive prompt stalls the headless run indefinitely and squats a slot; the only sanctioned escalation is `blocked = Yes` via the gated `update-status.sh --reason` path, which the concession-validator judges. One turn: execute every phase (Planning → Complete) without ending the turn between phases or writing hand-off messages; invoke each delegated skill by reading its SKILL.md or via the Skill tool. The ONLY acceptable turn ends: (a) `agent_status = Complete` with the final report delivered, or (b) `blocked = Yes` on a true-blocker per `yolo-true-blockers`. Resume does NOT decay this mode: re-entering an already-provisioned task (after a block, interruption, wake, or human poke) RE-ASSERTS full `--yolo` for the rest of the turn — a poke means "continue and finish", a previously-cleared block is not a reason to stop. About to end the turn with phases remaining and no fresh true-blocker? Continue instead.</rule>
<rule id="yolo-true-blockers">Even in `--yolo`, STILL pause and set `blocked = Yes` on Asana when any of these apply: (a) destructive op with no recovery path (force push outside a PR branch, git history rewrite on shared branch, file deletion outside scratch/build dirs); (b) user-only credential needed (2FA, password, OAuth re-auth, signing key passphrase); (c) no defensible default exists (genuine ambiguity that could flip task outcome wholesale); (d) risk of overwriting unstaged user work (dirty working tree on a non-agent-created branch). SETTING THE BLOCK IS GATED: write it via `~/.config/agent-watcher/update-status.sh <gid> <status> --blocked yes --reason "<the precise blocker>"` — the `--reason` is mandatory. A PreToolUse gate (`require-concession-validation.sh`) runs the `/concession-validator` on that reason against the true-blocker taxonomy; if the validator rules it NOT legitimate, you are NOT blocked — do exactly what its `what_to_try` says (swap-to-fund, build the missing maestro flow per `build-the-test-harness`, link the unmerged dep, attempt the action) and CONTINUE. A funds/provider block also requires a real attempt logged via `~/.config/agent-watcher/log-attempt.sh` (result `loss:`/`failed:`/`blocked:`) — predicting a wall is not hitting it. When `blocked = Yes` is set, do NOT attach a run-report doc (per `report-as-attachment`, docs are produced only at `Complete`). Set the `blocked` field and add at most ONE brief one-line Asana comment naming the blocker so a human knows what input is needed to unblock. The full run report is produced once, when the task ultimately reaches `Complete`. CARVE-OUT to (a): spending SANCTIONED test funds — the sim-testing playbook roster accounts, at the playbook's amounts — is NEVER condition (a), no matter how new or unvetted the code path is. A real test swap through prototype plugin code is the prescribed verification (build-and-test `gui-dependency-integration`, `test-drives-the-real-action`), not a destructive op. Do not invoke (a) to skip a funded in-app test. The ONLY funds-related true-blocker is an OBSERVED loss: an attempted swap/send that failed AND lost principal. Fees/slippage never count (budgeted at $15 equivalent per run, per the sim-testing playbook), and a loss can only be established by an actual attempt, never predicted.</rule>
<rule id="yolo-no-silent-downgrade">Finalizing — opening a PR or setting `Complete` — while the prescribed in-app drive did NOT reach success is itself a CONCESSION (a DOWNGRADE), gated exactly like a formal block. The SAME PreToolUse gate (`require-concession-validation.sh`) intercepts a `/pr-create` or `update-status.sh Complete` when your latest attempt-log entry is a wall (`failed:`/`loss:`/`blocked:`) or a `/tmp/agent-test-blocker-<gid>.md` note exists — i.e. you verified a weaker way, skipped the in-app drive, or bailed on a crash. It hands you the wall reason; run `/concession-validator <gid> "<reason>"` and obey the verdict. `legitimate:false` means the downgrade is NOT earned — do its `what_to_try` (apply the documented continue-workaround for a known gotcha like the Fabric `uiManagerDidDispatchCommand` SIGSEGV / maestro mis-targeting per the sim-testing playbook, swap-to-fund, build the missing harness) and reach the real success. The gated direct-verification fallback is legitimate ONLY with a GENUINE funded attempt logged (`log-attempt.sh` result `failed:`/`loss:`/`blocked:`) AND no remaining continue-workaround — this silent downgrade-to-fallback (the Xgram "stopped to avoid an accidental send" that never consulted the validator) is the exact bypass the gate was extended to catch. Do not pre-empt the gate by skipping the attempt; ATTEMPT, log, then let the verdict rule.</rule>
<rule id="yolo-stop-at-pr">In `--yolo` mode, NEVER merge the PR, tag a release, deploy, publish a package, or perform any other "land/ship" action. The agent's terminal action is reaching `agent_status = Complete` after the watch loop reports all-green. Merging is the human's decision. Force-pushing to the PR's own branch (to apply review/CI fixes) is allowed and expected. CARVE-OUT — a LAND/MERGE task: when the task's EXPLICIT deliverable is to land or merge an EXISTING PR (the task is about merging, not building a feature — signals: name/description like "land", "merge", "finish landing <X> PR", or it names a specific open PR to merge with no new code expected), the merge IS the deliverable, not a premature ship. Do NOT stop-at-PR and do NOT block on this rule; instead delegate to `/pr-land`, which by default enables GitHub auto-merge so the PR lands when CI is green (see pr-land). SECOND CARVE-OUT — landing an APPROVED feature PR at finalize-green, per `land-on-approval`. Everything else (a feature task with an unapproved PR and no Force Land field) still stops at PR. If genuinely ambiguous whether the task wants a merge, treat it as a normal feature task and stop at PR.</rule>
<rule id="land-on-approval">At finalize-green (and ONLY there — `finalize-gate` green means every reviewer-bot thread is already resolved, so auto-merge can never arm over an unaddressed automated review), decide landability before Complete: `landable = (the PR carries a human APPROVED review — ANY point in its history counts, it need not be on the current HEAD; comment-fix commits after the approval are fine because this run already addressed the threads and re-tested behavioral changes per `followup-reopens-status` (4)) OR (~/.cursor/skills/asana-force-land.sh <gid> prints land-approved — the operator's no-review override for trivial changes)`. If landable: run `/pr-land` passing the TASK URL, never the bare PR number — pr-land walks the task + subtasks to assemble the full multi-repo set, which is what triggers its dep ordering (defer-gui: merge dep → publish → upgrade-dep on develop → only then rebase+merge the GUI PR). pr-land owns the entire land contract: rebase, conflict resolution, verify, arming GitHub auto-merge, staging cherry-pick per `build-field-routing`. Never raw `gh pr merge`. A dep publish inside the land needs the operator's npm 2FA (pr-land `npm-otp-required`): that is a user-only credential — land everything landable before the publish boundary, then set `blocked = Yes` naming the OTP need (per `yolo-true-blockers` (b)); do not skip the publish or dismiss the remaining PRs. If pr-land reports `blocked` (changes requested) or `unsupported`, fall back to stop-at-PR and say why in the report. When the land armed or merged, SKIP `cheese-build-on-green` (CI builds wherever the landing happened) and set Complete with the report stating `merged` or `auto-merge armed, waiting on CI`. Not landable → stop at PR per `yolo-stop-at-pr`, exactly as before (a fresh task's first run is naturally not landable — this branch fires in practice on followups, after a human has approved).</rule>
<rule id="pr-watch-loop-amend-pattern">When iterating on PR feedback (CI failures, bugbot findings, etc.) inside the watch loop, prefer `git commit --amend --no-edit` + `git push --force-with-lease` over fixup commits. The PR's history should stay a single clean commit (or the minimum set of logically distinct commits the original implementation needed). Never use `--force` without `--with-lease` — if the branch has been touched by someone else, that's a true-blocker (set `blocked = Yes`).</rule>
<rule id="pr-watch-bounded-poll">Every step-6 wait is ONE call: `~/.cursor/skills/one-shot/scripts/watch-pr.sh --pr <num> --task-gid <gid>`. The script owns the budget arithmetic — it computes a single 30-minute deadline per task on first call and bounds each subsequent watch by the remaining budget. Exit contract: 0 = all checks pass; 1 = a check failed (fix, amend, re-run); 75/124 = budget exhausted (take the blocked=Yes exit). Never hand-roll a polling loop; never spawn anything to wait (`never-self-respawn`).</rule>
<rule id="never-self-respawn">In NO phase — Planning, Developing, Reviewing, Testing, or the step-6 watch — may you use `/loop`, `/schedule`, `ScheduleWakeup`, a background `claude &`, or `claude --resume`. Not to wait, not to re-check, and NOT as a "fallback in case X hangs" (e.g. a maestro or build capture). Concretely forbidden, not just `/loop`: `ScheduleWakeup` to heartbeat a long iOS build or Metro bundle ("wake me in 5 min in case the build stalls") is exactly this violation — the wake fires and re-injects the original `/one-shot` prompt. Bound the build with `timeout <seconds>` and react in THIS turn; do not schedule a safety wake. Every one of these re-invokes, resumes, or schedules another `claude`/`cli` process and can self-replicate into a fork storm. A scheduled wake also re-injects the original prompt, which restarts the whole task (see `ignore-refired-one-shot`). Any wait is a single blocking call in THIS process; the step-6 wait specifically follows `pr-watch-bounded-poll`. If a sub-operation might hang, bound it with `timeout <seconds>` — never schedule a wake to recover from it.</rule>
<rule id="report-as-attachment">Do NOT post progress, narrative, or status comments to the Asana task during the run (no per-phase updates, no "agent paused" essays). Run state is conveyed by the `agent_status`/`blocked` field transitions, nothing else. Attach the structured run-report DOC **only at `Complete`** — the true end of the run. A `blocked = Yes` is NOT terminal: the task may be unblocked and continue (see `finalize-gate`), so a block must NOT attach a doc (this is the spurious "doc on blocking" we want gone). On block, convey it with the `blocked` field plus at most ONE brief one-line Asana comment naming the blocker; produce the full report only once, at completion. At `Complete`: produce exactly ONE structured run report from the template `~/.cursor/skills/one-shot/templates/agent-run-report.md` — fill the frontmatter and every section (including the infra-capture fields: `sim_udid` from `$AGENT_SIM_UDID`, `metro_port` from `$AGENT_METRO_PORT`, `slot_index` from `node ~/.config/agent-watcher/lib/slots.js get --task-gid <gid>`; these let post-hoc evals verify resource accounting after the slot is released), using `_None observed._` for empty ones (never omit a section), keep it dense — and attach it via `asana-task-update.sh --task <gid> --attach-file <path> --attach-name agent-run-report.md`, with at most ONE Asana comment pointing to it. If no task GID is available (ad-hoc text task), skip the attachment and report only in chat. The chat-facing summary is unaffected. FRONTMATTER HONESTY: if `verified` is `not-run` or `partial`, `verify_blockers` MUST be non-empty and name why (`precondition` | `harness` | `code` | `task-drafting`); an empty `verify_blockers` paired with `verified: not-run`/`partial` is a silent downgrade-finalize that `require-concession-validation` now intercepts at `Complete` (per `yolo-no-silent-downgrade`), so fill it truthfully or reach `verified: pass` first.</rule>
<rule id="bugbot-in-watch">Treat the configured reviewer bots (`reviewer-bots`) as checks inside the step-6 watch — the watch blocks until their check-runs complete, and each force-push re-triggers them on the new HEAD. A reviewer-bot check-run that completes `NEUTRAL` (not `SUCCESS`) posted findings and is NOT clean — `gh pr checks`/`watch-pr.sh` will exit 0 on it (neutral is a non-failure), so you MUST separately check each reviewer bot's conclusion and review state, not rely on the watch's exit code (see `finalize-gate`). For `cursor[bot]` invoke `/bugbot` to examine and FIX findings whenever its check-run is `NEUTRAL` or it left a `COMMENTED` review, even though the watch passed; address other allowlisted bots inline. Do NOT arm bugbot's recurring cron (`CronDelete` it if armed) — the watch does the waiting. Reviewers can't reach clean within budget → `blocked = Yes` (no doc on block, per `report-as-attachment`).</rule>
<rule id="reviewer-bots">The automated PR reviewers to gate on live in asana-config `.watcher.reviewer_bots` (default `["cursor[bot]", "cursor"]` — Cursor's bot posts as `cursor[bot]` over REST and `cursor` over GraphQL; both are listed, but the load-bearing match is `__typename == "Bot"` per `finalize-gate`, not the literal logins). The step-6 watch and `finalize-gate` wait on EVERY one (its check-run completed-clean on HEAD + no unresolved threads from it). Additionally — semantic catch — if a check-run or PR review comes from an author that LOOKS like a bot (a `[bot]` login, or an automated/non-human reviewer) but is NOT in the allowlist: do not silently ignore it. If it posted actionable findings, treat the PR as not-green until addressed, and record "new automated reviewer `<login>` detected — consider adding to `.watcher.reviewer_bots`" under the run report's **Skill Gaps / Follow-ups & Risks** section. This keeps the allowlist authoritative while surfacing new reviewers for you to confirm.</rule>
<rule id="finalize-gate">The single definition of "done", owned by the CONTINUOUS monitor — never a one-off. Green = all CI checks pass (`gh pr checks <pr>`) AND every `reviewer-bots` check-run completed-clean on HEAD AND no unresolved threads from any of them — evaluated over the PRIMARY PR(s) only. CHECK UNRESOLVED THREADS THE RIGHT WAY: bot findings post as INLINE review threads, which `gh pr view --json reviews,comments` does NOT surface — using it makes the gate blind and is how a live cursor[bot] thread slipped past a Complete. Query review threads via graphql, fetching the author's `__typename` (NOT just `login`): `gh api graphql -f query='{ repository(owner:"O",name:"R"){ pullRequest(number:N){ reviewThreads(first:100){ nodes{ isResolved isOutdated comments(first:1){ nodes{ author{ login __typename } } } } } } } }'` and treat any `isResolved:false` thread whose first comment's author is a BOT as not-green. MATCH BOTS BY `__typename == "Bot"`, never by a literal login: GraphQL strips the `[bot]` suffix, so Cursor's reviewer appears as `cursor` (and its security reviewer also as `cursor`), while REST callers see `cursor[bot]` — a literal `cursor[bot]` comparison MISSES the `cursor` thread and silently passes its findings (the makeMaxSpend false-Complete: an unresolved HIGH-severity `cursor` thread slipped a Complete because the gate matched only `cursor[bot]`). PREFER the shared `pr-address.sh fetch` (or `pr-land-comments.sh`), which already do this `__typename`-based bot detection — do not hand-roll the literal-login comparison. `reviewer-bots` in asana-config lists both Cursor logins (`cursor[bot]`, `cursor`) as a defensive fallback, but the `__typename == "Bot"` check is the primary. Never substitute `gh pr view`'s reviews/comments for the reviewThreads check. NEUTRAL IS NOT CLEAN: `gh pr checks` buckets a reviewer-bot check-run conclusion of `NEUTRAL` as a non-failure, so the watch passes it — but `NEUTRAL` means the bot POSTED FINDINGS without hard-failing (Cursor Bugbot's neutral = "I found issues, non-blocking"). For a `reviewer-bots` author, "check-run completed-clean" means conclusion `SUCCESS` or `skipped` ONLY; a `NEUTRAL` conclusion, or a bot REVIEW in state `COMMENTED`/`CHANGES_REQUESTED`, means examine its findings and address/resolve them (cursor → `/bugbot`, others inline) before green — never treat the green `gh pr checks` exit as covering a reviewer bot. The unresolved-reviewThreads check above is the backstop that catches the findings; the NEUTRAL/COMMENTED state is the signal that triggers the examination. PRIMARY PR(s) means EVERY non-draft PR this run created or pushed to, in every repo — enumerate them explicitly before declaring green (e.g. the run's own PR list; `gh pr list --author @me` per touched repo as a cross-check) and gate on EACH. A PR created minutes before finalize (a second-repo PR, a late dep bump) is the one the gate exists to catch — gating only the first/oldest PR while a newer one's CI is still running is a completion-honesty violation. DRAFT dependency PRs (per `dep-pr-draft-vs-bump`) are EXCLUDED from the gate: a draft dep PR will fail CI/bugbot until its dep is published+bumped, so do NOT gate completion on it and do NOT run the reviewer-bot watch on it. Record each draft dep PR under the run report's **Follow-ups & Risks** as "un-draft + re-gate after `<dep>` publishes". The draft exclusion is for DEPENDENCY PRs ONLY — a PRIMARY PR (the run's actual deliverable) that is itself left in DRAFT can NEVER be green: reviewer bots do not run as check-runs on a draft, so a CI-only `watch-pr.sh` exit 0 covers CI alone and is NOT "every reviewer-bot completed-clean on HEAD." Treating a draft primary PR's CI-green as done — or dismissing its unresolved bot threads as "historical / commented, not check-runs" — is a completion-honesty violation. To finalize a primary PR: mark it ready-for-review (`gh pr ready <pr>`) so the bots re-run on HEAD, then gate normally; unresolved reviewer-bot threads must be replied/resolved (`/bugbot` for `cursor[bot]`) before Complete, regardless of whether they predate this run. When green AND a task GID with an `agent_status` field exists → set `agent_status=Complete` (`~/.config/agent-watcher/update-status.sh <gid> Complete`). This is exactly what one-shot step 6→7 runs, AND what `/bugbot` runs when invoked standalone with a task GID (to finish a task manually after a blocker has been cleared). `/pr-address` NEVER sets Complete — it's a one-off and automated reviews can land after it finishes; completion belongs to whatever monitors continuously.</rule>
<rule id="cheese-build-on-green">AFTER `finalize-gate` reports green and BEFORE attaching the run report or setting `Complete` (Complete triggers session retirement, so post-Complete work races the watchdog sweep): read the task's Build field — `~/.cursor/skills/asana-build-field.sh <gid>`. If it names a cheese (`feta|gouda|halloumi|cheddar`), kick the cheese build via /cheese: target branch `test-<value>`, source ref = the task's PR head branch. CHEESE IS A POINTER, NOT A WORKSPACE (cheese:`pointer-not-workspace`): all work lives on the PR branch and is already pushed there; the cheese op is ONLY a hard-reset of `test-<value>` to that pushed head + force-push — CI auto-builds on `test-*` pushes. Never commit to or develop on a `test-*` branch. Do NOT pin dep tarballs in orch runs — plain pointer reset only; if the run produced unmerged dep-repo PRs, note in the report that the cheese build excludes them until they publish or an operator pins them manually. Followups need no special casing: a re-engaged run that ends green re-runs the same reset, so the test branch tracks the amended head. `staging` routes through /pr-land (not this rule); `none` means no build routing. SKIP the cheese push entirely when this run is LANDING the PR per `land-on-approval` — CI builds wherever the landing happened, so a cheese build would be redundant. This `test-*` force-push is sanctioned and is NOT a ship action under `yolo-stop-at-pr`. Record the push (branch, source sha) in the run report's Summary.</rule>
<rule id="followup-reopens-status">A finished task (`agent_status = Complete`/`Archived`) is RE-ENGAGED by the operator setting it back to `Pending` — the SINGLE re-engage signal. The watcher then RESUMES your prior session (your full prior conversation restored, on a FRESH slot with a working sim + Metro) when a transcript exists, or fresh-spawns a never-run task. So when you find yourself resumed on a task you previously finished, you are continuing a FOLLOWUP: your prior context is intact, the status is already off `Complete` (the resume moved it), and your resources are live. Do NOT move the status yourself, and do NOT expect dead resources (the watcher's resume re-provisions a fresh slot, unlike the old in-place un-retire). Apply: (1) FOLLOWUP SCOPE — the operator's comments since your last run-report ARE the deliverable, per `followup-scope-is-the-deliverable`; (2) WORKTREE — if your prior worktree was pruned and the task has an OPEN PR, re-provision on that PR's branch via `~/.config/agent-watcher/setup-task-workspace.sh --task-gid <gid> --repo <name> --existing-branch <pr-head-branch>`, never a fresh branch off develop (that orphans the PR's work); (3) STALE-PRIOR-HISTORY GUARD — your prior report/conclusions are historical context, NOT current truth about the environment: re-verify every env/tooling claim (debug flags, dev servers, build paths, provider availability) before working around it ("X didn't work" before is not evidence X fails now); (4) PR-COMMENT-ADDRESS — if the task's open PR has any UNRESOLVED review threads (human OR reviewer-bot), addressing them IS part of this run's deliverable, with NO separate `/pr-address` invocation: reuse `/pr-address`'s reply-then-resolve workflow inline (read `~/.cursor/skills/pr-address/SKILL.md`) — for each unresolved thread apply the fix, reply stating what changed, then resolve the thread — but commit the fixes via THIS skill's `pr-watch-loop-amend-pattern` (amend + force-push), NOT fixup commits. Flow the phase statuses as you go: `Developing` (applying fixes) → `Testing` (re-run `/build-and-test`, since a comment fix can change behavior, per step 6 point 3) → `Reviewing` (re-push, re-gate). Both human and bot threads must be replied-and-resolved before green; one-shot is the continuous monitor that owns finalize, so IT — never `/pr-address`, per pr-address `never-sets-completion` — sets `Complete`. Then re-finalize per `finalize-gate`. Phase statuses (`Developing`/`Reviewing`/`Testing`) are progress indicators YOU set as you go, NOT re-engagement triggers — setting a finished task to a phase status does nothing (the watchdog's old un-retire-on-phase-status was removed 2026-06-25); `Pending` is how a finished task comes back.</rule>
<rule id="followup-scope-is-the-deliverable">A resume is a FOLLOWUP when the operator added new scope to the task AFTER the last `agent-run-report` was attached. That attachment is the "agent reported done here" WATERMARK: any operator comment NEWER than it — or, if the task has never had a report attached, EVERY operator comment — is scope this run has NOT yet discharged. BEFORE finalizing, ENUMERATE the operator comments since that watermark with the LIVE fetch `~/.config/agent-watcher/check-followup-scope.sh --task-gid <gid>` (it fetches the task's comments + attachments, computes the watermark from the latest `agent-run-report*.md` attachment's `created_at`, prints every newer comment, and writes the marker the Complete gate checks). RECALLED CONTEXT NEVER SATISFIES THIS: every watcher resume compacts your session from a summary built BEFORE the followup comment existed (the resume menu's "Resume from summary" runs /compact), so "the newest comment is my own Complete note" remembered from context is exactly the stale claim that re-Completed past the UTXO followup — only the script's live output counts, and the `require-followup-scope-on-complete.sh` PreToolUse gate blocks `update-status.sh <gid> Complete` without a fresh check. Treat each concrete ask (added workstreams, new acceptance criteria, "also do X") as THIS run's PRIMARY deliverable — NOT the task's pre-existing open PR. Re-anchoring on the already-open PR, verifying/finishing it, and setting `Complete` while that newer scope is deferred is a FALSE COMPLETE: it advances the report watermark PAST asks it never addressed, so they silently vanish — the exact miss that lost Workstreams A/B on the RN-upgrade task (the run shipped the base-upgrade PR, set Complete, and parked the asks in a run-report line with ZERO subtasks). DELIVER it: do the new scope to its DoD like any task and ship its PR(s) — to the parent, or, for work in ANOTHER repo, to a per-repo subtask per `multi-repo-subtasks`. Subtasks are UNWATCHED containers (no `agent_status`, never spawned by the watcher — the orch tracks ONLY the parent); a subtask exists to hold one extra-repo PR or split-out context, NOT to offload scope to a separate run. So you cannot discharge the new scope by "creating a subtask for it" — only by DELIVERING the work. If a newer ask is genuinely too large to finish in this run, that is established by ATTEMPTING it, NEVER by predicting from the code: you may surface it as unfinished ONLY after a GENUINE logged attempt (link the WIP/unpublished dep via updot/`--existing-branch`, pin its tarball, build, spike the integration) hit a concrete wall recorded via `log-attempt.sh` (`failed:`/`blocked:` with the real error). A predicted "too big / WIP-only / would require finishing an unmerged migration" without that logged spike is a PREMATURE descope, gated like any concession (the validator denies it as the `unmerged dependency` / `too-large-unattempted` deny-on-sight category). Having actually hit the wall: ship what you can, then SURFACE the remainder explicitly in one Asana comment that names exactly what is undone and why (so the operator can spin it into its own top-level task), and do NOT set the parent `Complete`. HARD GATE: do NOT attach the run-report and set the PARENT `Complete` while any operator ask newer than the prior report is neither DELIVERED (its work shipped, PR attached to the parent or a per-repo subtask) nor explicitly surfaced-as-unfinished with the parent left non-`Complete`. Finishing the pre-existing deliverable is fine and may earn its own PR, but it does NOT discharge the newer scope. DISTINGUISH from a same-deliverable resume (`followup-reopens-status`): a resume that merely FINISHES the SAME blocked deliverable with NO operator comment newer than the last report legitimately ends at `Complete`; the trigger is operator scope added since the watermark. "Documented as follow-ups" buried in a report line, or a placeholder subtask with no delivered work, is precisely the false-complete this forbids.</rule>
</rules>

<step id="1" name="Collect input">
Accept one of:

1. Asana task URL
2. Text/file requirements

**LAND/MERGE TASK ROUTING (check FIRST).** If the task's deliverable is to LAND or MERGE an existing PR rather than build something (per `yolo-stop-at-pr`'s land-task carve-out — name/description like "land", "merge", "finish landing <X> PR", or it points at a specific open PR with no new code expected), do NOT run the build flow (plan/im/pr-create). Move `agent_status` to `Reviewing`, run `/pr-land` against that PR (its default enables GitHub auto-merge so it lands when CI is green), then finalize: set `agent_status = Complete` once the PR is merged (or has auto-merge armed and is just waiting on green CI — say which in the report). This is the sanctioned land path in `--yolo`. Everything below is for normal build tasks.

**EXISTING-PR / COMMENT-ADDRESS ROUTING (check before building).** If this task already has an OPEN non-draft PR — a resume after a `Pending` bounce, OR a fresh-spawn whose prior transcript was pruned — do NOT rebuild from scratch and do NOT open a second PR. Detect it from the task's attached PR link (attached on the prior run per `attach-prs-by-default`), falling back to `gh pr list --state open --head "${GIT_BRANCH_PREFIX:-jon}/<short-name>"`. Re-provision on that PR's branch (`~/.config/agent-watcher/setup-task-workspace.sh --task-gid <gid> --repo <name> --existing-branch <pr-head-branch>`), then: if it has unresolved review threads, enter the comment-address path per `followup-reopens-status` (4) before re-gating; otherwise resume the watch per `finalize-gate`. This makes the comment-address loop correct whether or not the resume restored the prior transcript.

Optional flags:

- `--asana-task <gid>` (explicit Asana GID override)
- `--no-asana-attach` (opt OUT of the GitHub-widget attach step; attach is ON by default per `attach-prs-by-default`. Attach uses `ASANA_GITHUB_SECRET` from credentials.json and degrades to a warning if absent)
- `--yolo` (hands-off mode: defer soft questions to a final summary, only block on true-blockers — see `yolo-execution` and `yolo-true-blockers` rules)

**Per-task worktrees (you create them).** When the agent-watcher spawns this session as a parallel slot, the working directory is `~/git` — NOT a pre-made worktree. Once the plan (step 2) identifies the target repo(s), create a dedicated, co-located worktree for each repo this task will modify:

`~/.config/agent-watcher/setup-task-workspace.sh --task-gid <gid> --repo <name> --branch "${GIT_BRANCH_PREFIX:-jon}/<short-name>"` → prints the worktree path. Derive `<short-name>` as a short kebab-case slug from the task title (same convention as `/im`'s `$GIT_BRANCH_PREFIX/<short-description>`, e.g. `upgrade-piratechain-sdks`) — descriptive, NOT the opaque task GID. Use the SAME `<short-name>` branch across every repo of this task.

They land together under `~/git/.agent-worktrees/<task-gid>/<repo>/` on the branch you passed, off `origin/develop`, with `env.json` copied in and `node_modules` APFS-cloned, so tooling + secrets work without extra setup. `cd` into the PRIMARY repo's worktree and do all build/test/commit/push there. (Manual, non-watcher runs already sit in a normal `~/git/<repo>` checkout — skip this provisioning.)

**Editing an EdgeApp gui dependency (edge-core-js, edge-currency-accountbased, edge-exchange-plugins, edge-currency-plugins, edge-login-ui-rn, …).** Create a co-located worktree for each repo the task actually modifies, under the same `~/git/.agent-worktrees/<task-gid>/` dir. For a dependency-only task that's just the dep; for a task that also changes gui code, both. That's ALL one-shot does for deps — it does NOT touch `DEBUG_*`/`updot`/`env.json`. Linking the modified dep into the app and exercising it is **entirely `/build-and-test`'s job** (`gui-dependency-integration`), which creates the co-located `edge-react-gui` worktree itself if the task didn't already. (Run repo scripts with each repo's package manager per its lockfile — yarn is being phased out, don't assume.)
</step>

- Text/file mode: create plan from provided requirements

<step id="3" name="Implementation phase">

(Skip on manual non-watcher runs already inside a normal checkout.) Then set agent_status=Developing and run `/im` using the approved `/asana-plan` output.

<step id="4" name="Local verification — orchestration only">
**`--yolo` (orchestration) only — SKIP this step entirely on manual runs.** A human running `/one-shot` by hand verifies as they see fit; do NOT run `/build-and-test` for them. In `--yolo`: run `/build-and-test` on the implemented branch BEFORE opening the PR, so the PR opens already-verified. "Verified" means `/build-and-test` actually drove the **real end-to-end user action to its terminal success in the running app on the sim** per `test-on-sim-by-default` + `test-drives-the-real-action` — NOT `tsc`/build passing, NOT a precursor. A swap-plugin change is verified only when a swap actually EXECUTES in-app (the success scene), never at the quote; do every prerequisite (link the dep into core/gui, build, fund/switch accounts) to get there. Do NOT declare the Testing phase done by static reasoning, by "it builds", or by stopping short at a quote/partial step. Set agent_status=Testing at the start of this step (on-sim/local verification is its own phase per `agent-status-on-pending-task`, between Developing and the PR). No PR exists yet, so fixes amend the local commit (no force-push). If it fails, amend HEAD with the fix (`git commit --amend --no-edit`) and re-run; repeat up to 2 times. If still failing after 2 attempts, set `blocked = Yes` with the reason and stop — do NOT open a PR on red local verification.
</step>

<step id="5" name="PR phase">
Set agent_status=Reviewing. Then run `/pr-create` — always pass `--asana-task <gid>` (so the Asana link is embedded in the PR body, per `task-gid-for-pr-body-link`) AND `--asana-attach` (attach the PR to the task via the GitHub widget, per `attach-prs-by-default`), and never pass `--asana-assign`. If the run produced PRs in MORE THAN ONE repo, instead build the subtask-per-PR structure per `multi-repo-subtasks` (and `dep-pr-draft-vs-bump` for dependency PRs). The step-4 test run saved proof screenshots (`/tmp/agent-proof-<gid>-NN-<slug>.png`, per build-and-test's `proof-screenshots-for-pr`); `/pr-create` step 4b attaches them to the PR as the test-evidence comment — verify that happened (every gui-tested PR should carry its screenshots).

Task GID source priority: (1) explicit `--asana-task <gid>`; (2) Asana task URL from step 1; (3) chat context from prior steps.
</step>

<step id="6" name="PR watch (gate to Complete)">
Keep agent_status=Reviewing (set in step 5; the PR is up and under external review by CI, reviewer bots, and humans). Wait for external green signals before marking `Complete`. Status stays at `Reviewing` throughout. Every wait is one `watch-pr.sh` call per `pr-watch-bounded-poll`; the script owns the 30-minute budget.

1. **CI checks**: run `~/.cursor/skills/one-shot/scripts/watch-pr.sh --pr <pr-num> --task-gid <gid>`. When it returns —
   - exit 0 (all pass) → CI is green
   - exit 1 (a check failed) → read the failing job's log via `gh run view --log-failed`, apply a fix, then amend + force-push per `pr-watch-loop-amend-pattern`, then re-run `watch-pr.sh`
   - exit 75 or 124 (budget exhausted) → take the 30-min exit below
2. **Reviewer bots**: handled as part of the watch per `bugbot-in-watch` + `reviewer-bots`. The watch blocks until every allowlisted reviewer's check-run completes on HEAD; when it returns, if any is red or has unresolved threads, fix it (`/bugbot`'s scan/fix for `cursor[bot]`, inline for others), amend + force-push (re-triggers them), then re-run `watch-pr.sh`. Also apply the semantic-catch for non-allowlisted bots per `reviewer-bots`. Never arm bugbot's cron.
3. **Re-test after a fix (semantic)**: after applying ANY fix in this loop, reason about whether the change could affect behavior that `/build-and-test` covers. If it could (logic/UI/build changes) AND this is a `--yolo` run, re-run `/build-and-test` before re-entering the watch. For non-behavioral fixes (lint, comments, docs, pure config), skip the re-test. Don't blindly re-test every fix; don't skip it for real behavioral changes. When a re-test produced NEW proof screenshots, attach them to the PR via `pr-attach-screenshots.sh --title "Test evidence (after fix)"` so the evidence on the PR matches the current HEAD.

Exit conditions:
- **All green** (per `finalize-gate`: CI checks pass + every `reviewer-bots` check-run completed-clean on HEAD + no unresolved threads from any of them): proceed to step 7.
- **30 min wall-clock elapsed**: set `blocked = Yes` with a comment summarizing what was still red, then stop.
- **True-blocker hit during a fix attempt**: set `blocked = Yes` per `yolo-true-blockers`, stop.

Honor `yolo-stop-at-pr` strictly: never merge, never tag, never deploy DURING the watch. The only mutations here are force-pushes to the PR's own branch. After green, step 7's routing may sanction a land (`land-on-approval`, via /pr-land) or a `test-*` cheese push (`cheese-build-on-green`) — those are the only post-green exceptions.
</step>

<step id="7" name="Report (attach run report, then Complete)">
Build the run report and attach it, THEN mark Complete. Per `report-as-attachment`, this attachment (not comments) is how the run is documented.

0. Land + build routing FIRST (before the report, so the outcome is recorded in it):
   a. Landability per `land-on-approval`: check the PR for a human APPROVED review (`gh pr view <pr> --json reviews`) and run `~/.cursor/skills/asana-force-land.sh <gid>`. Landable → `/pr-land <task-url>` (task URL, never the bare PR), then skip (b).
   b. Not landing: cheese check per `cheese-build-on-green` — `~/.cursor/skills/asana-build-field.sh <gid>`; on a cheese value, run /cheese (`test-<value>` ← PR head, no dep pins). Then continue.
1. Copy `~/.cursor/skills/one-shot/templates/agent-run-report.md` to a scratch path named with BOTH the GID and the task short-name: `/tmp/agent-run-report-<gid>-<short-name>.md`. Fill the frontmatter — set `agent_session_uuid` from `$AGENT_SESSION_UUID` (the orchestration session that ran this), plus `outcome: complete`, `verified`, `verify_blockers`, repo/branch/pr, started/ended, `skills_used` — and every section. Use `_None observed._` for empty sections; keep it dense (bullets, signal over prose). Map content to sections:
   - phases that ran (`/asana-plan`, `/im`, `/pr-create`, `/build-and-test`) + watch-loop iteration counts → **Summary**.
   - the `/build-and-test` run as RUN EVIDENCE (do not leave thin — this is the visibility surface): the real action driven to terminal success (or the exact step it stopped at + why), method (static checks tsc/jest/eslint/verify-repo with results, sim build flavor, maestro flow path + MCP-explore vs yaml-proof, any `/debugger` use), environment (sim UDID/slot, roster account + any `env.json` switch, funding asset+amount + any swap-to-fund, provider forced + reverted), proof-screenshot paths + confirmation they are PR-attached, any gated direct-verification fallback, and what stayed unverified (mirror `verify_blockers`) → **Testing**.
   - in `--yolo`, every auto-deferred decision (question, default chosen, reversibility) → **Decisions**.
   - build/test/debug learnings → **Dev Notes & Gotchas** (inline-tagged). Harness friction → **Orchestration**. Skill defects → **Skill Gaps**. Missing/weak task inputs → **Task-Drafting Feedback**.
2. Attach it: `asana-task-update.sh --task <gid> --attach-file /tmp/agent-run-report-<gid>-<short-name>.md --attach-name agent-run-report-<short-name>.md`. (Optionally one pointer comment.) Skip if no task GID (ad-hoc task) and report in chat only.
3. Set the `tested` field to the method(s) actually performed (per `set-tested-field`): `~/.config/agent-watcher/set-tested.sh <gid> <Option ...>`.
4. Set agent_status=Complete — ONLY after `finalize-gate` reports all-green.
5. Return a short chat summary + PR URL + phases ran.

Per `report-as-attachment`, the run-report doc is attached **only at `Complete`**. On `blocked = Yes`, do NOT attach a doc — set the `blocked` field + at most one brief one-line comment naming the blocker (`yolo-true-blockers`); the report is produced once, at completion.
</step>
